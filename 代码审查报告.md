## 二、P0 严重问题（必须修复）

### P0-1 协程作用域泄漏：PlaybackService 滥用 `MainScope().launch`

- **文件**：`app/src/main/java/com/lossless/music/player/PlaybackService.kt`
- **行号**：87, 207, 300, 311

PlaybackService 在 4 处使用 `kotlinx.coroutines.MainScope().launch { ... }`。`MainScope()` 每次都创建一个**独立、无法取消**的 CoroutineScope，Service 销毁时这些协程仍在运行，造成协程泄漏；若协程内引用了已释放的 repository，会引发崩溃。

```kotlin
// 第87行 - 泄漏示范
currentSongId?.let { sid ->
    kotlinx.coroutines.MainScope().launch {  // 每次新建 scope，无法取消
        repository.clearResumePosition(sid)
    }
}
```

**修复**：定义 Service 级 scope，`onDestroy` 中取消：
```kotlin
private val serviceScope = CoroutineScope(Dispatchers.Main + SupervisorJob())
// 用 serviceScope.launch { ... }
override fun onDestroy() {
    serviceScope.cancel()
    // ...
}
```

### P0-2 静态变量持有 Service 实例，内存泄漏 + 竞态

- **文件**：`app/src/main/java/com/lossless/music/player/PlayerController.kt`
- **行号**：68（设置）、148（清空）、196（声明）、160/174（轮询）

```kotlin
companion object {
    var service: PlaybackService? = null  // 静态引用持有 Service
}
```

- `PlayerController` 是 `@Singleton`，生命周期等同 Application。静态变量持有 Service 引用，Service 被系统重建时旧引用未清空，调用其方法会崩溃。
- `skipToNext()`（行 160）用 `while (service == null && retries < 30) { delay(100) }` 轮询等待，是典型竞态条件，不可靠。

**修复**：移除静态引用，统一通过已建立的 `MediaController` 作为 UI 与 Service 唯一通信桥梁。

### P0-3 反射访问 ExoPlayer 私有字段，极度脆弱

- **文件**：`app/src/main/java/com/lossless/music/player/SleepTimerManager.kt`
- **行号**：133-153

```kotlin
private fun setPlayerVolume(vol: Float) {
    PlayerController.service?.let { svc ->
        try {
            val field = svc.javaClass.getDeclaredField("player")  // 反射
            field.isAccessible = true
            val player = field.get(svc) as? ExoPlayer
            player?.volume = vol
        } catch (_: Exception) {}  // 静默吞异常
    }
}
```

字段改名/类型变更即刻失效，异常被完全吞掉。且 `loadAndPlay()`（行 299-306）中响度归一化也会设 `p.volume`，**两者相互覆盖**，渐弱退出功能失效。

**修复**：在 PlaybackService 暴露公开方法 `setVolume(Float)` / `getVolume()`，避免反射。

### P0-4 `player!!` 强制非空，并发场景 NPE 风险

- **文件**：`app/src/main/java/com/lossless/music/player/PlaybackService.kt`
- **行号**：241-243

```kotlin
if (mode == PlayMode.REPEAT_ONE && current != null && player != null) {
    player!!.seekTo(0)   // 判空后到使用之间 player 可能被 onDestroy 置空
    player!!.play()
    return
}
```

`player` 是可变 `var`，并发场景下判空后到使用之间可能被置空，`!!` 抛 NPE。

**修复**：用局部变量 `val p = player ?: return`。

### P0-5 QueueManager 状态变量无同步保护，竞态条件

- **文件**：`app/src/main/java/com/lossless/music/player/QueueManager.kt`
- **行号**：39, 46-47, 114-158

`QueueManager` 是 `@Singleton`，被 PlaybackService 和多个 ViewModel 共享。`currentIndex`、`shuffleOrder`、`shufflePointer` 是普通可变变量，无同步。`next()` 先读 `_playMode`、再读 `_current`、再改 `currentIndex`，非原子。UI 调 `jumpTo()` 的同时 Service 调 `next()`，会导致索引错乱、跳歌或越界。

**修复**：队列操作方法加 `@Synchronized`，或用单线程调度器（Mutex）。

---

### P0-6 OnlineLyricsFetcher 在主线程执行同步网络请求

- **文件**：`app/src/main/java/com/lossless/music/ui/lyrics/OnlineLyricsFetcher.kt`
- **行号**：30-47

`fetchLyrics` 不是 `suspend`，内部 `httpGet` 用 `HttpURLConnection` 同步阻塞。调用方 `PlayerViewModel.loadLyrics()` 在 `viewModelScope.launch`（默认主线程）中调用，会**阻塞主线程导致 ANR**。

**修复**：改为 `suspend fun fetchLyrics(...)`，内部 `withContext(Dispatchers.IO) { ... }`。

### P0-7 onDestroy 中 player 释放逻辑缺陷，mediaSession 为 null 时 player 不释放

- **文件**：`app/src/main/java/com/lossless/music/player/PlaybackService.kt`
- **行号**：147-157

```kotlin
override fun onDestroy() {
    mediaSession?.run {
        player.release()   // mediaSession 为 null 时此块不执行
        release()
    }
    player = null          // 字段置空，但原 ExoPlayer 实例泄漏！
}
```

若 `mediaSession` 为 null，`player` 字段指向的 ExoPlayer 不会被 `release()`，造成 native 资源泄漏。

**修复**：
```kotlin
mediaSession?.release()
player?.release()
player = null
```

### P0-8 AndroidManifest 使用错误的 MediaButtonReceiver 包名

- **文件**：`app/src/main/AndroidManifest.xml`
- **行号**：54-60

```xml
<receiver android:name="androidx.media.session.MediaButtonReceiver" .../>
<!--                     ^^^^ 这是 media1 旧包，项目用的是 Media3 -->
```

项目用 Media3（`androidx.media3.session`），但声明的 `MediaButtonReceiver` 来自旧的 `androidx.media.session`（media1），两者不兼容，**耳机线控/蓝牙按键事件无法正确路由**。

**修复**：改为 `androidx.media3.session.MediaButtonReceiver`，或直接移除（Media3 MediaSessionService 已内置处理 MEDIA_BUTTON）。

## 五、架构层面总结

1. **Service 与 UI 通信混乱**：同时存在静态引用（`PlayerController.service`）和 MediaController 两套机制，应统一走 MediaController。
2. **PlayerViewModel 职责过多**：既管播放又管歌词又管收藏，可拆分。
3. **Flow 收集**：`HomeViewModel` 的 `allSongs`/`favorites` 等是冷 Flow，未 `stateIn` 缓存，切换 Tab 反复订阅。
4. **协程生命周期管理**：整个 player 模块对协程作用域管理薄弱（P0-1、P1-1、P1-13），是系统性缺陷。
5. **线程安全**：单例的可变状态普遍无同步（P0-5、P1-4、P1-5），是系统性缺陷。

---

## 六、修复优先级建议

**立即处理（影响崩溃/功能）**：P0-1、P0-2、P0-3、P0-6、P0-8
**发布前处理（影响稳定性）**：P0-4、P0-5、P0-7、P1-8、P1-9、P1-2
**迭代优化**：其余 P1、P2

整体而言，项目功能完整度高、注释详尽、分层基本清晰，但在**协程生命周期管理**、**线程安全**、**Service 与 UI 通信机制**三方面存在系统性缺陷，建议发布前集中重构 player 模块通信层。
